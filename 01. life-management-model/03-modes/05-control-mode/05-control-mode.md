# Control mode

## Inbox
* **див документи в папці і підпапках**
* чи є реальні результат
* Це повторюється?
* системні проблеми не “вирішуються”. вони контролюються
* 👉 спочатку шукаю pattern → тільки потім думаю сам
* 👉 якщо проблема не вирішується за N сесій:
→ піднятись на рівень вище
→ або змінити протокол

### 🔁 ESCALATION

👉 якщо проблема не вирішується:
- піднятись на рівень вище
- змінити протокол


---

## АЛГОРИТМ

1. перевірка
2. виявлення проблем
3. класифікація
4. реакція

- всі проблеми фіксуються

# 🛠 КРОК 1 — КОНТРОЛЬ СИСТЕМИ (DIAGNOSTIC)

👉 Мета: перевірити не життя, а **як працює система управління**

---

## 🔗 ЛАНЦЮГ (швидка перевірка)

- [ ] стратегія → місяць
- [ ] місяць → тиждень
- [ ] тиждень → день
- [ ] день → дія
- [ ] дія → результат

👉 якщо є розрив → ТИП ЗБОЮ: РОЗРИВ МІЖ РІВНЯМИ

---

## 1. ПЕРЕВІРКА (2–3 хв)

### РУХ
- [ ] Я реально працював над реалізацією орієнтирів (не тільки думав)

👉 якщо ні → ТИП ЗБОЮ: НЕДОСТАТНІЙ РУХ

---

### ФОКУС
- [ ] Я працював над ключовими орієнтирами
- [ ] Не було розпорошення

👉 якщо ні → ТИП ЗБОЮ: РОЗФОКУС

---

### ТРАНСЛЯЦІЯ
- [ ] Місяць → відповідає стратегії
- [ ] Тиждень → рухає місяць
- [ ] День → рухає тиждень

👉 якщо ні → ТИП ЗБОЮ: РОЗРИВ МІЖ РІВНЯМИ

---

### ВИКОНАННЯ
- [ ] Я реально виконував задачі
- [ ] Я не зависав в аналізі

👉 якщо ні → ТИП ЗБОЮ: СЛАБКЕ ВИКОНАННЯ

---

### TIMEBOX
- [ ] Я дотримався timebox
- [ ] Я не затримувався в IMPROVE

👉 якщо ні → ТИП ЗБОЮ: SYSTEM FREEZE

---

### РЕЗУЛЬТАТ
- [ ] Є відчутний прогрес
- [ ] Реальність змінюється

👉 якщо ні → ТИП ЗБОЮ: НЕПРАЦЮЮЧА СТРАТЕГІЯ

---

### НАВАНТАЖЕННЯ  
- [ ] Я не перевантажив систему  
- [ ] Кількість активних цілей відповідає capacity  

👉 якщо ні → ТИП ЗБОЮ: OVERLOAD

## 🧠 PRIORITY ENGINE

👉 Мета: визначити ОДИН дефект, над яким працюю зараз

---

### 1. ОЦІНКА ДЕФЕКТІВ

Для кожного активного дефекту:

IMPACT (1–5):  
FREQUENCY (1–5):  
BLOCKING (0–2):  

PRIORITY SCORE = IMPACT + FREQUENCY + (BLOCKING × 2)

---

### 2. ВИБІР

👉 Головний дефект = максимальний PRIORITY SCORE

👉 Працюю тільки з ним (макс 1–2 активних)

---

### 3. ПРАВИЛА ПРІОРИТЕТУ (override)

❗ BLOCKING ALWAYS WINS  
якщо дефект блокує виконання → він головний незалежно від score

---

❗ EXECUTION > EVERYTHING  
якщо немає виконання → не лагоджу нічого іншого

---

❗ OVERLOAD FIRST  
якщо перевантаження → спочатку зменшую load

---

❗ STRATEGY LAST  
якщо немає результату, але execution слабкий → не змінюю стратегію

# Коли control вважається завершеним 

- визначено 1 основний дефект
- для нього є чітка дія або орієнтир
- визначено тип (оперативний / системний)
- не більше 1–2 активних дефектів

❗ якщо дефектів більше → неправильна сесія
