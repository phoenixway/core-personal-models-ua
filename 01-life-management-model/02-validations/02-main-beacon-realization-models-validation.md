# ✅ VALIDATION — МОДЕЛІ РЕАЛІЗАЦІЇ ГОЛОВНИХ ОРІЄНТИРІВ

👉 Мета:
швидко перевірити, чи для кожного головного орієнтиру визначено достатньо ясну і придатну до використання модель реалізації

---

## 1. ЧИ ПОКРИТІ ВСІ ГОЛОВНІ ОРІЄНТИРИ?

- [ ] для кожного головного орієнтиру визначено модель реалізації
- [ ] немає головних орієнтирів, які “висять” без способу втілення
- [ ] не залишилось орієнтирів у стані “важливо, але незрозуміло як реалізується”

👉 якщо ні → рівень невалідний

---

## 2. ЧИ ВИЗНАЧЕНО ТИП РЕАЛІЗАЦІЇ?

Для кожного головного орієнтиру:

- [ ] визначено тип реалізації
- [ ] тип вибраний не інтуїтивно, а за реальною механікою втілення
- [ ] зрозуміло, чи це:
  - стратегічний
  - проектний
  - підтримуючий
  - змішаний

👉 якщо ні → рівень невалідний

---

## 3. ЧИ Є МОДЕЛЬ РЕАЛІЗАЦІЇ ЯК МЕХАНІКА?

Для кожного головного орієнтиру:

- [ ] модель реалізації описує, через що саме орієнтир втілюється
- [ ] це не просто красива декларація
- [ ] це не список задач
- [ ] це не готова стратегія, а саме спосіб реалізації

👉 якщо ні → рівень невалідний

---

## 4. ЧИ ЗРОЗУМІЛО, ЩО Є РЕАЛЬНОЮ РЕАЛІЗАЦІЄЮ?

Для кожного головного орієнтиру:

- [ ] я можу назвати, що буде вважатися реальною реалізацією
- [ ] це описано не абстрактно, а як зрозумілий результат або стан
- [ ] можна відрізнити "реалізується" від "просто думаю про це"

👉 якщо ні → рівень невалідний

---

## 5. ЧИ ВИЗНАЧЕНО, ЧИ ПОТРІБНЕ СТРАТЕГІЧНЕ ПРОЕКТУВАННЯ?

Для кожного головного орієнтиру:

- [ ] зрозуміло, чи потрібне окреме стратегічне проектування реалізації
- [ ] стратегічне проектування не призначається автоматично всім підряд
- [ ] проектні орієнтири не були штучно протягнуті через стратегічну машину
- [ ] підтримуючі орієнтири не були замасковані під стратегічні

👉 якщо ні → рівень невалідний

---

## 6. ЧИ ВИЗНАЧЕНО ОСНОВНИЙ МЕХАНІЗМ РЕАЛІЗАЦІЇ?

Для кожного головного орієнтиру:

- [ ] зрозуміло, чи він реалізується:
  - через довгострокову стратегію
  - через окремий проект
  - через підтримуючий контур
  - через комбінацію
- [ ] є явна передача цього орієнтиру в наступний релевантний контур

👉 якщо ні → рівень невалідний

---

## 7. ЧИ ВІДФІЛЬТРОВАНО ПОМИЛКИ МОДЕЛІ?

- [ ] модель реалізації не плутається з головним орієнтиром
- [ ] модель реалізації не плутається з тижневими / денними задачами
- [ ] модель реалізації не дублює довгострокову стратегію
- [ ] немає явно зайвих або надуманих моделей

👉 якщо ні → рівень невалідний

---

## 8. ЧИ Є FREEZE?

- [ ] результат сесії зафіксовано
- [ ] моделі реалізації не допилюються без кінця
- [ ] після сесії можна рухатись далі в стратегічне проектування або довгострокову стратегію

👉 якщо ні → рівень невалідний

---

## 9. DONE CONDITION

Рівень вважається достатньо готовим якщо:

- [ ] для кожного головного орієнтиру є модель реалізації
- [ ] для кожного орієнтиру зрозумілий тип реалізації
- [ ] для кожного орієнтиру зрозуміло, що вважати реальною реалізацією
- [ ] зрозуміло, чи потрібне стратегічне проектування
- [ ] зрозуміло, в який контур цей орієнтир передається далі
- [ ] цього достатньо, щоб не вигадувати механіку заново щодня

---

## 10. УМОВНА ГОТОВНІСТЬ

Можна позначити як УМОВНО ГОТОВО якщо:

- [ ] основна логіка реалізації вже зібрана
- [ ] лишилися не критичні уточнення
- [ ] немає блокування подальших рівнів
- [ ] є DEADLINE доопрацювання

👉 у такому випадку:
УМОВНО ГОТОВО ДО: [Дата]
