## КРОК 1 — ВИБІР ОРІЄНТИРІВ

Взяти список головних орієнтирів із [[00-main-beacons]]

Для кожного орієнтиру окремо відповісти:

```text
цей орієнтир взагалі як реалізується?
```

---

## КРОК 2 — ВИЗНАЧЕННЯ ТИПУ РЕАЛІЗАЦІЇ

Для кожного головного орієнтиру визначити базовий тип реалізації:

- стратегічний
    
- проектний
    
- підтримуючий
    
- змішаний
    

---

### ЗМІСТ ТИПІВ

#### стратегічний

Орієнтир реалізується через:

- довгі напрямки
    
- стратегічні рішення
    
- розподіл ресурсів
    
- життєву оркестрацію
    

#### проектний

Орієнтир реалізується через:

- окремий проект
    
- набір фаз
    
- завершуваний результат
    

#### підтримуючий

Орієнтир реалізується через:

- регулярне підтримання стану
    
- правила
    
- стандарти
    
- періодичний контроль
    

#### змішаний

Орієнтир потребує:

- стратегічного контуру
    
- і окремих проектів / підтримуючих механізмів
    

---

### RULE

```text
тип реалізації визначається не за красою формулювання,
а за реальною механікою втілення
```

---

## КРОК 3 — ПЕРЕВІРКА НА РЕАЛЬНУ МЕХАНІКУ

Для кожного орієнтиру відповісти:

```text
через що саме цей орієнтир втілюється?
```

Питання-підказки:

- це втілюється через довгу зміну життя?
    
- через окремий проект?
    
- через регулярне підтримання?
    
- через комбінацію цих способів?
    

---

## RULE

```text
якщо не можна пояснити через що саме це реалізується,
модель реалізації ще не визначена
```

---

## КРОК 4 — ФОРМУЛЮВАННЯ МОДЕЛІ РЕАЛІЗАЦІЇ

Для кожного головного орієнтиру записати:

```text
МОДЕЛЬ РЕАЛІЗАЦІЇ:

цей орієнтир реалізується через ...
```

---

### ПРИКЛАДИ

- "залишити слід" реалізується через стратегічне створення цінного публічного внеску
    
- "заробіток" реалізується через стратегічний розвиток джерел доходу
    
- "закрити балкон склом" реалізується через окремий проект
    
- "здоров’я" реалізується через підтримуючий контур регулярного забезпечення
    

---

## КРОК 5 — ВИЗНАЧЕННЯ КРИТЕРІЮ РЕАЛІЗАЦІЇ

Для кожного головного орієнтиру відповісти:

```text
що буде вважатися реальною реалізацією цього орієнтиру?
```

---

### RULE

```text
реалізація = не абстрактне "рухаюсь",
а зрозумілий тип результату або стану
```

---

### ПРИКЛАДИ

- "залишити слід" → існують створені й передані назовні цінні результати
    
- "заробіток" → є стабільний і зростаючий дохід
    
- "балкон" → балкон фізично засклений
    
- "здоров’я" → стан стабільно підтримується на визначеному рівні
    

---

## КРОК 6 — ЧИ ПОТРІБНЕ СТРАТЕГІЧНЕ ПРОЕКТУВАННЯ

Для кожного орієнтиру відповісти:

```text
чи потребує цей орієнтир окремого стратегічного проектування реалізації?
```

Варіанти:

-  так
    
-  ні
    
-  частково
    

---

### RULE

```text
стратегічне проектування потрібне,
якщо реалізація залежить від факторів, вибору напрямків, trade-offs і довгої логіки руху
```

---

## КРОК 7 — ВИЗНАЧЕННЯ ОСНОВНОГО МЕХАНІЗМУ РЕАЛІЗАЦІЇ

Для кожного орієнтиру зафіксувати:

- через стратегію
    
- через проект
    
- через підтримку
    
- через комбінацію
    

---

## КРОК 8 — ФІЛЬТР

Відкинути або переписати моделі, якщо:

- вони надто абстрактні
    
- вони не описують механіку
    
- вони плутають орієнтир із задачею
    
- вони не дають зрозуміти, як це входить у систему
    

---

### BOUNCE

```text
якщо:
- занадто абстрактно → конкретизувати механіку
- це вже задача → опустити нижче
- це вже стратегія → спростити до моделі реалізації
```

---

## КРОК 9 — FREEZE

Результат фіксується:

- моделі реалізації не оптимізуються далі в цій сесії
    
- використовуються як є
    
- передаються далі в:
    
    - [[02-main-beacon-strategic-projecting]] (де потрібно)
        
    - [[03-long-term-strategy]]
        

---

### ВИНЯТОК

Перегляд тільки якщо:

- змінено головні орієнтири
    
- з’явилися нові важливі дані
    
- control показав системну помилку моделі реалізації
    

---

# 📦 ВИХІД

Для кожного головного орієнтиру:

- тип реалізації
    
- модель реалізації
    
- критерій реальної реалізації
    
- чи потрібне стратегічне проектування
    
- основний механізм реалізації
    

---

# ✅ ВАЛІДНІСТЬ

Моделі реалізації валідні якщо:

- для кожного головного орієнтиру визначено модель реалізації
    
- зрозуміло, через що саме орієнтир реалізується
    
- визначено тип реалізації
    
- визначено, що вважати реальною реалізацією
    
- зрозуміло, чи потрібне стратегічне проектування
    
- усе сформовано в межах timebox
    

---

# ⚠️ ПРАВИЛА

1. головний орієнтир ≠ модель його реалізації
    
2. модель реалізації має описувати механіку, а не красиву декларацію
    
3. не всі орієнтири потребують стратегічного проектування
    
4. проектне не треба насильно тягнути в стратегічну машину
    
5. підтримуюче не треба маскувати під проект
    
6. не оптимізувати після freeze
    
7. модель реалізації не замінює довгострокову стратегію, а готує вхід для неї
    

---

# ⚡ КОРОТКО

1. взяти головні орієнтири
    
2. для кожного визначити тип реалізації
    
3. описати механіку реалізації
    
4. визначити, що є реальною реалізацією
    
5. визначити, чи потрібне стратегічне проектування
    
6. freeze

# 10. Найкоротша формула всього цього

> **Орієнтир сам по собі нічого не робить.  
> Потрібна модель його реалізації.  
> Модель має спускатися в дорожню карту і далі в action units.**

І ще одна:

> **Не кожному орієнтиру потрібен однаковий сценарій.  
> Але кожному потрібен свій механізм перетворення в реальність.**
