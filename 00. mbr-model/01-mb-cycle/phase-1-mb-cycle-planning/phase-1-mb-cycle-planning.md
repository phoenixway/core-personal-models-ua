## Затверджений алгоритм
### 0. Початкові умови

- Є множина `активних орієнтирів = {O1, O2, …, On}`
- Є важливі специфічні "технічні" орієнтири - власне уточнення і пріоритезація головних орієнтирів
-  Є обмежений ресурс: час тижня
### 1. Ініціалізація циклу

#### Встановити загальний статус реалізації орієнтирів
- Оцінити загальний стан головних орієнтирів. 
- Встановити необхідність "технічних" орієнтирів [[02. mb-clarification]] і [[03. mb-prioritization]]
#### Встановити індивідуальні статуси орієнтирів
- Сформулювати і записати всі затверджені `активні орієнтири`.
- Кожен орієнтир крім специфічних **в кожен момент має рівно один статус**:
    - `УТОЧНЕННЯ`
    - `ПРОЕКТУВАННЯ`
    - `РЕАЛІЗАЦІЯ`
    - `РЕФЛЕКСІЯ`
- Додаткові статус
    - inactive — орієнтир існує лише декларативно
	- unsupported — є намір, але нема систем
	- fragile — система є, але нестабільна
	- active — стабільна реалізація
	- saturated — орієнтир не потребує активного фокусу
- Для кожного орієнтиру, крім "технічних", визначити поточний статус `Si`.
- Якщо `статус` незрозумілий:  `Si` = УТОЧНЕННЯ

>**Правило: статус орієнтира не змінюється всередині тижня**

### 2. Перевірка керованості

- Порахувати кількість орієнтирів `|активних орієнтирів|`.
- Якщо  `|активних орієнтирів|` > 5:     

>**заморозити найменш актуальні орієнтири, щоб |актуальних орієнтирів| := 3–5`

- Перевірити: чи для кожного `Oi` зрозуміло, **який тип роботи він потребує**.
- Якщо для якогось `Oi` тип роботи незрозумілий:     

>**Si := УТОЧНЕННЯ**`

### 3. Розподіл часу і енергії між орієнтирами

- Встановити сумарний бюджет часу на цикл `T`.
- Перш ніж планувати подвиги, чесно оціни свій рівень енергії (0–100%) `En`.
- Для кожного орієнтиру `Oi` задати **вагу** `Wi` (не обов’язково точну):
```
- високий пріоритет → більша вага
- підтримуючий орієнтир → менша вага
```
- встановити співвідношення в відсотках між необхідним уточненням головних орієнтирів і роботою над уже відомими орієнтирами
- Нормалізувати ваги так, щоб `Σ Wi = 100% часу і сил`
```
Це - орієнтовний розподіл, не контракт.
```
#### Якщо енергії < 30%
- Активуй **Режим Енергозбереження**.
- Обери **лише один** найважливіший Орієнтир на цей цикл.
- Всі інші Орієнтири постав на "Паузу".
- Плануй тільки мінімально необхідні дії та сон.
### 4. Для кожного орієнтиру призначення фази реалізації 

- Для кожного `Oi` в залежності від статусу:

| **Статус**   | **Фаза**                                |
| ------------ | --------------------------------------- |
| УТОЧНЕННЯ    | [[phase-1-specific-beacon-definition]]  |
| ПРОЕКТУВАННЯ | [[phase-2-specific-beacon-projecting]]  |
| РЕАЛІЗАЦІЯ   | [[phase-3-specific-beacon-realization]] |
| РЕФЛЕКСІЯ    | [[phase-4-specific-beacon-control]]     |
| ЗМІШАНА      | поєднання фаз                           |
### 5. Формування беклогів циклу

Для кожного `Oi`:
- Створити беклог `Bi` відповідно до `Si`. Сформулювати **НЕОБХІДНИЙ МІНІМУМ дій**, що відповідають його статусу і ситуації.

-  Якщо неможливо сформулювати дії:     

> 	`Si` := УТОЧНЕННЯ     
> 	перейти до Стадії 4`

- Перевірка виконуваності:  
> 	якщо дія виглядає як подвиг - **спростити дію**

- Беклог **не список бажань**, а:
    - впорядкована черга
    - перший елемент = **блокуюча дія**
        
- Виконується **не весь беклог**, а **стільки, скільки дозволено виділеним часом**
#### Якщо статус `УТОЧНЕННЯ`

Беклог містить:
- питання
- формулювання
- прояснення меж
- виключення хибних напрямків
    
Перший елемент:
> «Яке питання зараз блокує рух?»

####  Якщо статус `ПРОЕКТУВАННЯ`

Беклог містить:
- структуру цілі
- цикли
- метрики
- ініціативи
- спрощення

Перший елемент:
> «Яке рішення зробить реалізацію неминучою?»

#### Якщо статус `РЕАЛІЗАЦІЯ`

Беклог містить:
- конкретні виконувані дії
- без аналізу і змін цілі

Перший елемент:

> «Яка дія зараз фізично просуває стан світу?»

#### Якщо статус `РЕФЛЕКСІЯ`

Беклог містить:
- аналіз результатів
- корекції
- рішення про зміну статусу

Перший елемент:
> «Що показав цей цикл насправді?»

### 6. Валідація беклогів

Для кожного `Bi`:

`if перший елемент неконкретний або занадто великий:     розбити або переформулювати`

`if беклог виглядає нескінченним:     застосувати правило 80/20`
